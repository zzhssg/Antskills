# 三方职责分离 + Production 规则

## 三方职责矩阵

每个功能点都有明确的职责归属：

| 职责 | Backend（工程师） | AI（Claude / LLM） | Frontend（渲染层） |
|------|-------------------|--------------------|--------------------|
| API 调用 & 认证 | ✅ 负责 | ❌ | ❌ |
| 数据缓存 & 更新 | ✅ 负责 | ❌ | ❌ |
| 评分算法 | ✅ 实现 | 设计公式 | ❌ |
| 主观判断 & 洞察 | ❌ | ✅ 负责 | ❌ |
| 异常检测规则 | ✅ 实现 | ✅ 定义阈值 | ❌ |
| 异常叙事 | ❌ | ✅ 负责 | ❌ |
| UI 组件渲染 | ❌ | ❌ | ✅ 负责 |
| 用户输入验证 | ✅ 后端校验 | ❌ | ✅ 前端即时反馈 |
| 错误展示 | ❌ | ❌ | ✅ 负责 |

### 关键边界

1. **AI 不做数学** — L3 接收的是 L2 已经算好的结构化数据（含评分、排名、异常标签），AI 只负责把这些数据翻译成人话
2. **Frontend 不做计算** — L4 接收的是 L2 的数据 + L3 的文案，只负责灌入容器渲染
3. **Backend 不写文案** — 所有用户可见的文字（推荐理由、风险提示、异常说明）都由 AI 产出

## Production-vs-Prototype 二元规则

前端 Demo 和生产实现有本质区别：

| 维度 | Demo（原型） | Production（生产） |
|------|-------------|-------------------|
| 数据 | Mock / 程序生成 | 真实 API |
| 计算 | 在前端 JS 里 | 在后端 Python/Go |
| LLM 调用 | 直接 fetch API | 通过后端代理 + 缓存 |
| 错误处理 | console.log | 结构化错误码 + 用户友好提示 |
| 认证 | 无 | API Key 加密存储 |
| 并发 | 单用户 | 多用户 + 限流 |

**规则**：
- Demo 的**视觉**和**交互逻辑**是参考标准
- Demo 的**数据获取方式**和**计算位置**不是参考标准
- implementation-guide.md 必须明确标注哪些 Demo 行为是 prototype-only

## Retry Contract（错误重试契约）

当任何层级出错时，用户界面必须提供：

### 全局错误卡

```
┌─────────────────────────────────────────┐
│  ⚠ 数据获取失败                          │
│                                         │
│  [数据源名称] 暂时不可用                    │
│  最后成功时间: 2024-01-15 14:30 UTC       │
│                                         │
│  [🔄 重试]   [使用缓存数据]               │
└─────────────────────────────────────────┘
```

### 规则

1. **任何错误都不能让页面白屏** — 至少显示错误卡
2. **重试按钮**必须在首次失败后出现
3. **重试间隔**：1s → 3s → 10s → 停止自动重试，保留手动按钮
4. **缓存降级**：如果有缓存数据，必须提供「使用缓存数据」选项，并标注缓存时间
5. **部分失败**：如果 5 个数据源中 3 个成功 2 个失败，展示成功部分 + 失败部分的错误卡

### 每层降级方案必须在 implementation-guide.md 中明确列出
