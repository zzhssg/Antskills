# 底层方法论 — 核心原则

## 1. 金字塔原则（Pyramid Principle）

所有 Skill 的信息架构必须遵循金字塔结构：

```
          ┌─────────────┐
          │   结论/推荐   │  ← Hero 区：3 秒内理解
          ├─────────────┤
          │  论据/可视化  │  ← 中间层：支撑结论的数据
          ├─────────────┤
          │ 原始数据/方法 │  ← 信任层：可折叠
          └─────────────┘
```

**反模式**：先展示一堆数据，最后才给结论。这是「倒金字塔」，用户看到一半就走了。

**正确做法**：Hero 区直接说「推荐 A，因为 X」，然后用图表证明 X，最后把原始数据放在可折叠区域。

## 2. 价值感 → 信任感管线

用户的心理路径是：

```
看到结论 → "这对我有用吗？"（价值感）
  → 看到论据 → "凭什么这么说？"（信任感）
    → 看到方法论 → "我信了"（决策）
```

**每一层缺失都会中断管线。** 只有结论没有论据 = 空口白话；只有数据没有结论 = 信息垃圾。

## 3. 四层 Pipeline 分离原则

| 原则 | 含义 |
|------|------|
| **L1 只负责获取** | 不做任何计算、过滤、排序。只做 API 调用 → 统一 schema 映射 |
| **L2 只负责计算** | 纯函数，相同输入必须产生相同输出。不调 API、不调 LLM |
| **L3 只负责判断** | 接收 L2 的结构化数据，产出人话结论。不做数学计算 |
| **L4 只负责渲染** | 接收 L2 数据 + L3 文本，灌进容器。不做计算、不做判断 |

**为什么分离？**
- L1 出问题 → 只需要修 API 调用，不影响算法
- L2 出问题 → 可以用单测快速定位
- L3 出问题 → 调 Prompt 就行，不碰代码
- L4 出问题 → 改 CSS/组件，不碰逻辑

## 4. 降级优雅原则

每一层都必须有降级方案：

| 层 | 正常 | 降级 |
|----|------|------|
| L1 | 主数据源 | Fallback 数据源 → 缓存 → 空数据 + 错误卡 |
| L2 | 完整计算 | 跳过可选字段 → 用默认值 → 标注 "数据不完整" |
| L3 | LLM 调用 | Fallback 模板函数 → 纯数据卡（无文案） |
| L4 | 完整渲染 | 隐藏缺失组件 → 显示骨架屏 → 全局错误卡 |

## 5. 风险对称原则

**有推荐就必须有风险提示。** 推荐和风险必须在视觉上等权重呈现。

反模式：大字推荐 + 小字风险 → 误导用户。
正确做法：推荐卡下方紧跟同等视觉权重的风险卡。

## 6. 机会成本思维

用户不仅想知道「这个好不好」，更想知道「如果不选这个，我会错过什么」。

每个推荐都应包含：
- **正向**：选它能得到什么（APR、收益）
- **反向**：不选它会怎样（市场均值对比、次优选项差距）

## 7. 异常叙事化原则

数据异常不能只打标签（"⚠ 异常"），必须翻译成人话：

❌ `APY_OUTLIER: true`
✅ "这个产品的年化收益率 (28.5%) 是同类的 4.7 倍，其中 82% 来自代币激励。如果代币价格下跌 50%，实际收益率将降至约 6.1%。"

## 8. 数据来源透明原则

每个数据点必须可追溯：
- 数据来源（哪个 API / 哪个协议）
- 更新时间（UTC 时间戳）
- 计算方法（如果是衍生指标，说明公式）

## 9. 不做交易代理

Skill 的边界是**决策辅助**，不是交易执行。

- ✅ 可以：推荐产品、对比方案、模拟收益、风险分析
- ❌ 不可以：一键下单、自动转账、代签合约
