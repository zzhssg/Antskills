# S1 — 需求采集

## 目标

产出一份完整的**需求画布**，所有后续阶段都基于此画布展开。

## 方法论

**追问到画布填满为止。** 不允许在画布有空白时进入 S2。

## 需求画布模板（7 题）

### Q1. WHO — 用户是谁？
- 用户画像（经验、知识水平、决策习惯）
- 用户痛点（现在怎么做的？哪里不爽？）
- 典型使用场景（什么时候打开这个 Skill？）

### Q2. WHAT — 做什么？
- 核心功能（一句话描述）
- 关键输入（用户需要提供什么？）
- 关键输出（用户期望看到什么？）

### Q3. WHY — 为什么做？
- 现有替代方案是什么？
- 比替代方案好在哪？
- 不做的后果是什么？

### Q4. WHERE — 数据从哪来？
- 主数据源（API / MCP / 爬虫 / 手动输入）
- 辅助数据源
- 数据刷新频率要求

### Q5. WHEN — 时效性要求
- 实时 / 准实时（< 1min）/ 定期（小时/天）？
- 用户可接受的最大延迟？

### Q6. HOW — 交互形式
- 一次性结果 vs 持续监控？
- 主动推送 vs 被动查询？
- 移动端 / 桌面端 / 两者？

### Q7. PARADIGM — 范式选择 ⭐ v3 新增
- (a) Skill 本身就是最终产品，直接运行 → **A 实现型**
- (b) 需要产出规范文档，指导工程师开发 → **B 规范型** ⭐默认
- (c) 两者都要 → **C 双模**

## 执行步骤

1. 向用户展示需求画布模板
2. 用户回答后，检查是否所有 Q 都有实质性回答
3. 对模糊回答追问（例："数据从 API 来" → "具体哪些 API？有 Key 吗？频率限制？"）
4. 填满画布后，复述一遍确认
5. 用户确认 → 进入 S2

## 追问清单

如果用户的回答不够具体，用以下问题追问：

| 模糊回答 | 追问 |
|----------|------|
| "各种交易所" | 具体哪几家？优先级？ |
| "收益率数据" | APY / APR / 7日年化？含不含挖矿奖励？ |
| "实时的" | 5秒 / 1分钟 / 5分钟 可以接受？ |
| "安全的就行" | 审计报告？TVL 门槛？黑历史排除？ |
| "好看就行" | 有参考的网站/App 截图吗？ |
